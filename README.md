# nefor-team

Командная конфигурация поверх движка [nefor](https://github.com/amenocturne/nefor).
Превращает nefor в агентного ассистента, заточенного под нашу инфраструктуру:
авторизация через DP, модели через Nestor, Jira/Confluence-инструменты и
управляемый рабочий процесс с lead-оркестратором.

## Как это работает

Вы описываете задачу на естественном языке, а lead-оркестратор сам выбирает
граф субагентов: исследует код, предлагает план, ждёт одобрения для записывающих
ролей и затем делегирует выполнение.

| Роль     | Что делает                                      |
| -------- | ----------------------------------------------- |
| explorer | Только читает и исследует кодовую базу          |
| worker   | Выполняет одобренные изменения                  |
| reviewer | Только читает и ревьюит результат               |
| docs     | Работает с Jira/Confluence/документацией        |
| critic   | Только читает и критикует сложный план          |

Типичный цикл:

1. Вы описываете задачу.
2. Lead отправляет `explorer` изучить код, если задача не очевидна.
3. Для сложной задачи lead составляет план, прогоняет `critic` и дорабатывает план.
4. Lead показывает план через `write-review`.
5. Вы пишете `/approve` или `/reject` с комментарием.
6. `worker` или `docs` выполняют одобренные изменения, `reviewer` проверяет результат.

Роли только для чтения (`explorer`, `reviewer`, `critic`) запускаются без
одобрения плана. Записывающие роли (`worker`, `docs`) запускаются только после
`/approve`. После одобрения обычные file edit/write-инструменты внутри этих
ролей не требуют нового одобрения на каждую правку.

В `/safe` `dispatch-graph` и неоднозначные shell-команды остаются под проверкой
во время выполнения. В `/auto` всё, что потребовало бы ручного решения,
отклоняется с подсказкой как восстановиться. В `/yolo` все инструменты
одобряются автоматически.

## Установка и синхронизация

Нужны [Homebrew](https://brew.sh), `just` (`brew install just`) и
[DP CLI](https://devplatform.pages.devplatform.tcsbank.ru/spirit-user-docs/docs/cli/)
с выполненной авторизацией.

Версия движка закреплена в корневом `.env`:

```sh
NEFOR_VERSION=0.3.0
```

Единственный источник истины для установленной командной конфигурации — этот
репозиторий. Синхронизация намеренно перезаписывает `~/.config/nefor` из
`starter/` и копирует туда `.env`:

```sh
git clone https://gitlab.tcsbank.ru/crit-autoloans/nefor nefor-team
cd nefor-team
just sync
```

`just sync`:

- читает `NEFOR_VERSION` из `.env`;
- просит явное подтверждение `y/yes`;
- устанавливает ровно закреплённую версию nefor;
- удаляет и заново копирует `~/.config/nefor` из `starter/`;
- копирует `.env` в `~/.config/nefor/.env`;
- не запускает тесты и не изменяет файлы репозитория.

`just install` оставлен как alias на `just sync`. Старый `just copy` есть только
для ручного копирования конфигурации; для нормального обновления используйте
`just sync`.

При старте `starter/init.lua` сравнивает `nefor.version` с
`NEFOR_VERSION`: сначала из `.env` рядом с активным config, затем из
родительской `.env`, если локальный файл отсутствует или пуст. Установленная
конфигурация продолжает использовать `~/.config/nefor/.env`; `just run` из
репозитория берёт корневой `.env`. Если версии не совпадают, запуск
останавливается с подсказкой выполнить `just sync`.

## Запуск

```sh
# После `just sync`
nefor
```

Для тестирования без установки конфигурации:

```sh
# Из директории репозитория
just run
```

Для локальных приватных вариантов используйте файлы, которые git игнорирует:

- `.env.local` — локальные переменные окружения; `just run` загружает его после `.env`.
- `starter/config/local.lua` — локальные варианты `NEFOR_CONFIG` и хуки запуска провайдера.

`starter/config/local.lua` должен вернуть таблицу `variants`. Первым аргументом
в загружаемый фрагмент передаются вспомогательные функции: `all_roles`,
`workflow_with`, `shared_tool_gate`, `env_or`.

```lua
local h = ...

return {
  variants = {
    local_provider = {
      provider = {
        kind = "local",
        name = "local-provider",
        model = os.getenv("NEFOR_LOCAL_MODEL"),
      },
      tool_gate = h.shared_tool_gate,
      workflow = h.workflow_with(h.all_roles(os.getenv("NEFOR_LOCAL_MODEL"))),
      log_level = "info",
      spawn_provider = function(ctx)
        ctx.actor.spawn(require("compositors.provider").spawn_spec(
          ctx.provider_name,
          {
            ctx.bin(os.getenv("NEFOR_LOCAL_PROVIDER_BIN")),
            "--name", ctx.provider_name,
            "--model", ctx.chosen_model,
          },
          { agentic_loop = ctx.agentic_loop }
        ))
      end,
    },
  },
}
```

Запуск:

```sh
NEFOR_CONFIG=local_provider just run
```

## Выбор модели

По умолчанию используется модель, отмеченная `is_default` в Nestor.
Переключить можно через `/model` в чате или переменную окружения:

```sh
NEFOR_TEAM_MODEL=tgpt/some-other-model nefor
```

## Команды в чате

| Команда    | Что делает                                                |
| ---------- | --------------------------------------------------------- |
| `/approve` | Одобрить предложенный план                                |
| `/reject`  | Отклонить план с пояснением                               |
| `/model`   | Без аргумента — пикер моделей; с аргументом — переключить |
| `/new`     | Новый чат: очищает транскрипт и создаёт новую сессию      |
| `/resume`  | Восстановить предыдущую сессию                            |
| `/safe`    | Безопасный режим: approve/deny/defer через popup          |
| `/auto`    | Автономный режим: approve/deny без popup                  |
| `/yolo`    | Полное автоодобрение всех инструментов                    |
| `/help`    | Справка                                                   |
| `@путь`    | Прикрепить файл к сообщению через автокомплит             |

## Горячие клавиши

| Клавиша       | Действие                                           |
| ------------- | -------------------------------------------------- |
| `Ctrl+O`      | Развернуть/свернуть детали рассуждения и вызовов инструментов |
| `Ctrl+B`      | Показать/скрыть сайдбар с графом агентов           |
| `Esc`         | Прервать текущий ход; double-Esc прерывает всё     |
| `Enter` / `A` | Одобрить инструмент в popup разрешений             |
| `D` / `Esc`   | Отклонить инструмент в popup разрешений            |

## Разрешения

По умолчанию nefor работает в `/safe`: спрашивает перед записывающим
`dispatch-graph` и неоднозначными `bash`, а пользователь может одобрить,
отклонить или отложить решение через popup. Исследования только для чтения не
требуют одобрения плана. Запуск `worker` и `docs` требует одобренный план;
дальше одобрение распространяется на обычные правки в рамках этого плана.

`/auto` предназначен для автономных запусков: безопасные действия продолжают
автоодобряться, но всё, что в `/safe` ждало бы человека, отклоняется с
сообщением с подсказкой по восстановлению. `/yolo` полностью одобряет все
инструменты. `/safe` возвращает интерактивный режим.
