# nefor-team

Командная конфигурация поверх движка [nefor](https://github.com/amenocturne/nefor).
Превращает nefor в агентного ассистента, заточенного под нашу инфраструктуру
(авторизация через DP, модели через Nestor).

## Как это работает

Вы описываете задачу на естественном языке, а lead-оркестратор сам выбирает
граф субагентов: исследует код, предлагает план, ждёт одобрения для записывающих
ролей и затем делегирует выполнение.

| Роль     | Что делает                                      |
| -------- | ----------------------------------------------- |
| explorer | Read-only исследование кодовой базы             |
| worker   | Выполняет одобренные изменения                  |
| reviewer | Read-only ревью результата                      |
| docs     | Jira/Confluence/документация, может писать docs |
| critic   | Read-only критика сложного плана до выполнения  |

Типичный цикл:

1. Вы описываете задачу.
2. Lead отправляет explorer'ов изучить код, если задача не очевидна.
3. Для сложной задачи lead составляет план, прогоняет critic и дорабатывает план.
4. Lead показывает план через `write-review`.
5. Вы пишете `/approve` (или `/reject` с комментарием).
6. `worker`/`docs` выполняют одобренные изменения, `reviewer` проверяет результат.

Read-only роли (`explorer`, `reviewer`, `critic`) запускаются без одобрения плана.
Записывающие роли (`worker`, `docs`) запускаются только после `/approve`. После
одобрения обычные file edit/write инструменты внутри этих ролей не требуют нового
одобрения на каждую правку. В `/safe` `dispatch-graph` и неоднозначные
shell-команды остаются runtime-gated; в `/auto` то, что потребовало бы ручного
решения, отклоняется с подсказкой, как восстановиться; в `/yolo` все инструменты
одобряются автоматически.

## Установка и синхронизация

Нужны [Homebrew](https://brew.sh), `just` (`brew install just`) и
[DP CLI](https://devplatform.pages.devplatform.tcsbank.ru/spirit-user-docs/docs/cli/)
(установить и авторизоваться).

Версия движка закреплена в корневом `.env`:

```sh
NEFOR_VERSION=0.3.0
```

Единственный источник истины для установленной командной конфигурации — этот
репозиторий. Синхронизация намеренно перезаписывает `~/.config/nefor` из
`starter/` и копирует туда `.env`:

```sh
git clone https://gitlab.tcsbank.ru/crit-autoloans/nefor nefor-team
cd nefor-team
just sync
```

`just sync`:

- читает `NEFOR_VERSION` из `.env`;
- просит явное подтверждение `y/yes`;
- устанавливает ровно закреплённую версию nefor;
- удаляет и заново копирует `~/.config/nefor` из `starter/`;
- копирует `.env` в `~/.config/nefor/.env`;
- не запускает тесты и не изменяет файлы репозитория.

`just install` оставлен как alias на `just sync`. Старый `just copy` есть только
для ручного копирования конфигурации; для нормального обновления используйте
`just sync`.

При старте `starter/init.lua` сравнивает `nefor.version` с
`NEFOR_VERSION`: сначала из `.env` рядом с активным config, затем из
родительской `.env`, если локальный файл отсутствует или пуст. Установленная
конфигурация продолжает использовать `~/.config/nefor/.env`; `just run` из
репозитория берёт корневой `.env`. Если версии не совпадают, запуск
останавливается с подсказкой выполнить `just sync`.

## Запуск

```sh
# После `just sync`
nefor
```

Для тестирования без установки конфигурации:

```sh
# Из директории репозитория
just run
```

## Выбор модели

По умолчанию используется модель, отмеченная `is_default` в Nestor.
Переключить можно через `/model` в чате или переменную окружения:

```sh
NEFOR_TEAM_MODEL=tgpt/some-other-model nefor
```

## Команды в чате

| Команда    | Что делает                                                |
| ---------- | --------------------------------------------------------- |
| `/approve` | Одобрить предложенный план                                |
| `/reject`  | Отклонить план с пояснением                               |
| `/model`   | Без аргумента — пикер моделей; с аргументом — переключить |
| `/new`     | Новый чат (очищает транскрипт, создаёт новую сессию)      |
| `/resume`  | Восстановить предыдущую сессию                            |
| `/safe`    | Безопасный режим: approve/deny/defer через popup (по умолчанию) |
| `/auto`    | Unattended режим: approve/deny без popup; defer превращается в deny |
| `/yolo`    | Полное автоодобрение всех инструментов                    |
| `/help`    | Справка                                                   |
| `@путь`    | Прикрепить файл к сообщению (автокомплит)                 |

## Горячие клавиши

| Клавиша       | Действие                                          |
| ------------- | ------------------------------------------------- |
| `Ctrl+O`      | Развернуть/свернуть детали (thinking, tool calls) |
| `Ctrl+B`      | Показать/скрыть сайдбар с графом агентов          |
| `Esc`         | Прервать текущий ход (double-Esc — прервать всё)  |
| `Enter` / `A` | Одобрить инструмент в popup'е разрешений          |
| `D` / `Esc`   | Отклонить инструмент в popup'е                    |

## Разрешения

По умолчанию nefor работает в `/safe`: спрашивает перед write-capable
`dispatch-graph` и неоднозначными `bash`, а пользователь может approve/deny/defer
через popup. Read-only исследования не требуют одобрения плана. Запуск `worker`
и `docs` требует одобренный план; дальше одобрение распространяется на обычные
правки в рамках этого плана.

`/auto` предназначен для unattended-запусков: безопасные действия продолжают
автоодобряться, но всё, что в `/safe` ждало бы человека, отклоняется с recovery
сообщением. `/yolo` полностью одобряет все инструменты. `/safe` возвращает
интерактивный режим.
